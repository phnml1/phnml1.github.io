Overview

What this project solves.

  • 취업 지원 현황, 전형 일정, 면접 질문, 알림을 한곳에서 관리하는 취업 준비용 PWA
  • 여러 채용 플랫폼에 흩어진 지원 상태와 일정을 수동으로 추적해야 하는 문제를 해결
  • 사용자는 지원 목록, 전형 타임라인, 마감 일정, 알림을 한 화면 흐름 안에서 확인할 수 있음
Impact

Results worth noticing.

Result 01

전형 상태 동기화 규칙을 UI 컴포넌트 밖의 순수 함수로 분리하고 단위 테스트로 검증 가능한 구조를 만듦

Result 02

상세 수정 후 전체 refetch에 의존하지 않고 변경된 필드만 React Query 캐시에 반영하는 기준을 세움

Result 03

파일 업로드처럼 클라이언트가 최종 URL을 알 수 없는 경우는 invalidate를 선택해 캐시 정확성을 우선함

Result 04

FCM, Service Worker, PWA 설치 상태, iOS Safari 제약을 하나의 권한 요청 흐름으로 연결

Key Features

Product surface.

지원 현황 목록과 복합 필터

커서 기반 무한 스크롤과 전형/제출/결과 필터로 지원 데이터가 많아져도 탐색 흐름을 유지

지원 상세 타임라인

서류, 기타 전형, 면접, 최종 결과를 하나의 전형 흐름으로 묶어 현재 상태를 빠르게 파악

전형 결과 자동 동기화

불합격 발생/해제에 따라 최종 결과와 현재 단계를 함께 갱신해 화면 간 상태 불일치를 줄임

React Query 캐시 부분 갱신

수정 응답과 입력값으로 상세 캐시의 변경 범위만 갱신해 불필요한 전체 상세 재조회를 피함

FCM 기반 푸시 알림

Service Worker, FCM 토큰, foreground/background 메시지를 연결해 일정과 알림 확인 흐름을 지원

iOS/PWA 알림 권한 분기

iOS Safari, 설치된 PWA, 일반 브라우저, 권한 거부 상태를 나누어 권한 요청 UX를 다르게 처리

Technical Highlights

Problems, decisions, outcomes.

Case 01

문제: 전형 결과가 바뀔 때 현재 단계와 최종 결과가 화면별로 달라질 수 있음

문제: 전형 결과가 바뀔 때 현재 단계와 최종 결과가 화면별로 달라질 수 있음

  • 해결 방식: timelineSyncRules에서 실패 여부, 이전 단계 실패, 최고 통과 단계 계산을 순수 함수로 분리
  • 결과 구조: 서류/기타/면접 수정 훅에서 같은 동기화 규칙을 재사용하고, 주요 상태 전이 케이스를 단위 테스트로 검증
Case 02

문제: 상세 수정 후 전체 상세 데이터를 다시 가져오면 변경 범위보다 네트워크 비용이 커짐

문제: 상세 수정 후 전체 상세 데이터를 다시 가져오면 변경 범위보다 네트워크 비용이 커짐

  • 해결 방식: queryClient.setQueryData로 문서 전형, 면접 일정, 기타 일정, 알림 읽음 상태의 캐시 일부만 갱신
  • 결과 구조: 서버 응답만으로 안전하게 갱신 가능한 수정은 부분 갱신, 파일 URL처럼 서버 재생성이 필요한 수정은 invalidate로 분기
Case 03

문제: 지원 목록은 데이터가 늘어날수록 초기 로딩과 탐색 비용이 커짐

문제: 지원 목록은 데이터가 늘어날수록 초기 로딩과 탐색 비용이 커짐

  • 해결 방식: useInfiniteQuery와 IntersectionObserver를 조합하고 필터 조건을 queryKey에 포함
  • 결과 구조: 필터 변경 시 목록 캐시가 조건별로 분리되고, 다음 페이지는 커서 기준으로 이어서 로드
Case 04

문제: 브라우저와 설치 상태에 따라 Web Push 지원 조건이 다름

문제: 브라우저와 설치 상태에 따라 Web Push 지원 조건이 다름

  • 해결 방식: deviceDetect로 iOS, standalone PWA, Web Push 지원 여부를 판별하고 권한 모달 흐름을 분기
  • 결과 구조: iOS Safari는 PWA 설치 안내, 권한 거부 상태는 설정 안내, 허용 상태는 FCM 토큰 갱신으로 분리
Case 05

문제: 브라우저에서 백엔드 API 주소와 인증 전달 방식을 직접 다루면 보안과 응답 처리 책임이 섞임

문제: 브라우저에서 백엔드 API 주소와 인증 전달 방식을 직접 다루면 보안과 응답 처리 책임이 섞임

  • 해결 방식: Next.js Route Handler 기반 BFF(/api/bff)와 clientApi/serverApi를 분리
  • 결과 구조: 클라이언트 요청은 BFF를 통해 백엔드로 프록시되고, 빈 응답/에러 응답 처리 규칙을 공통화
Tech Stack

Why these tools were used.

Next.js App Router

페이지 라우팅, 인증 영역 분리, BFF Route Handler 구성을 한 프로젝트 안에서 관리하기 위해 사용

TypeScript

지원 상태, 전형 단계, 일정 결과처럼 enum 성격이 강한 데이터를 타입으로 제한하기 위해 사용

TanStack Query

서버 상태 캐싱, 무한 스크롤, mutation 이후 상세 캐시 부분 갱신을 처리하기 위해 사용

Zustand

전형 타임라인 UI 상태, 필터 패널, 신규 지원 등록 단계 입력값처럼 서버 데이터와 분리된 클라이언트 상태를 관리하기 위해 사용

React Hook Form + Zod

프로필, 게시글, 질문, 지원 등록 입력값을 폼 단위로 검증하기 위해 사용

Firebase Cloud Messaging + Service Worker

foreground toast와 background browser notification을 모두 처리하기 위해 사용

Radix UI + Tailwind CSS

접근성 기반 UI primitive와 프로젝트 스타일을 조합해 모달, 탭, 셀렉트, 토글을 일관되게 구현하기 위해 사용

MSW

백엔드 의존 없이 지원 상세, 알림, 커뮤니티 등 주요 API 흐름을 재현하기 위해 사용

Focus

What I focused on.

  • 사용자가 보는 전형 상태와 서버 데이터가 어긋나지 않도록 상태 전이 규칙을 먼저 정리
  • invalidateQueries와 setQueryData를 무조건적으로 쓰지 않고, 수정 데이터의 신뢰 범위에 따라 선택
  • 알림 권한 요청을 기능 구현이 아니라 브라우저별 UX 문제로 보고 iOS/PWA 조건을 분리
  • 도메인 로직을 컴포넌트에 숨기지 않고 features 단위의 hooks, lib, stores로 분리
  • 로딩, 에러, 빈 상태, 모바일 필터 패널을 함께 구현해 실제 사용 흐름이 끊기지 않도록 구성
Trade-offs

Limits and decisions.

  • 상세 페이지가 아직 Client Component 중심이라 App Router의 서버 렌더링 이점을 충분히 활용하지 못함
  • React Query 기본 staleTime과 gcTime이 0이라 페이지 이동 시 재요청이 쉽게 발생할 수 있음
  • 신규 지원 등록은 단계별 상태와 DTO 변환 구조가 있으나, 최종 제출 연결은 코드상 보완 여지가 있음
  • 알림 기능은 브라우저별 분기 구조는 갖췄지만, 실제 기기별 수신 성공률 측정 자료는 별도로 남아 있지 않음
  • 성능 개선 수치는 측정 가이드만 있고 before/after 로그가 없어 포트폴리오 본문에서는 수치로 주장하지 않음
Portfolio Summary

How to read this project.

  • Career Hub는 취업 지원 현황과 전형 일정을 한곳에서 관리하는 Next.js 기반 PWA입니다.
  • 지원 상세의 핵심은 전형 결과 변경 시 현재 단계와 최종 결과가 함께 맞춰지는 상태 동기화 구조입니다.
  • 이 규칙을 timelineSyncRules로 분리하고 주요 전이 케이스를 테스트해 UI와 도메인 로직의 결합을 줄였습니다.
  • 상세 수정 후에는 전체 재조회 대신 React Query 캐시의 변경 범위만 갱신하도록 구현했습니다.
  • 파일 업로드처럼 응답 재생성이 필요한 경우는 invalidate로 분기해 캐시 정확성을 우선했습니다.
  • FCM과 Service Worker를 연결하고 iOS Safari, 설치된 PWA, 일반 브라우저에 따라 알림 권한 UX를 다르게 처리했습니다.
  • 단순 CRUD보다 상태 일관성, 캐시 갱신 기준, 브라우저 제약 대응을 중심으로 설계한 프로젝트입니다.
