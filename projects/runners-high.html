Project Case Study

러너스하이

위치 기반 실시간 러닝을 지원하는 모바일 러닝 앱

Period2025.02 - 2025.07
Team졸업작품 팀 프로젝트 (FE 1 / BE 2)
Role프론트엔드 개발
React NativeExpoTypeScriptExpo RouterZustandTanStack QueryWebSocket(STOMP)expo-location
Overview

What this project solves.

  • 위치 기반 실시간 러닝을 지원하는 모바일 러닝 앱
  • 혼자 달리기, 코스 기반 경쟁, 크루 러닝의 상태 흐름을 한 앱에서 처리
  • 사용자는 현재 위치, 코스, 진행도, 경쟁자와의 거리 차이를 보며 러닝할 수 있음
Impact

Results worth noticing.

Result 01

FE 1명으로 지도, 러닝 모드, 실시간 위치 동기화, 크루 관리 화면을 구현

Result 02

러닝 모드별 커스텀 훅 구조로 상태 전이와 UI 분기 책임을 분리

Result 03

STOMP 재연결, heartbeat, 모드별 publish/subscribe 분기로 위치 동기화 흐름을 안정화

Result 04

솔로 러닝 페이스 계산을 최근 10초 거리 기반으로 구성해 GPS speed 직접 의존도를 낮춤

Key Features

Product surface.

지도 기반 코스 탐색

현재 지도 영역과 위치를 기준으로 주변 러닝 코스를 조회해 러닝 진입 흐름을 줄임

솔로 러닝 기록

GPS 좌표를 누적해 거리, 페이스, 경로 데이터를 러닝 기록으로 저장

과거 기록 경쟁

과거 러너의 진행률과 현재 진행 거리를 비교해 실시간 경쟁 경험 제공

크루 러닝

STOMP 기반 위치 공유로 크루원의 준비 상태와 러닝 중 위치를 동기화

위치 검색

Google Places API로 사용자가 원하는 장소 주변의 코스를 탐색 가능

인증 유지

Axios 인터셉터와 refresh token 저장 구조로 401 발생 시 토큰 갱신 후 요청 재시도

Technical Highlights

Problems, decisions, outcomes.

Case 01

문제: 솔로, 코스 경쟁, 크루 러닝의 상태 전이가 한 흐름에 섞이면 UI 분기와 기록 저장 로직이 복잡해짐

문제: 솔로, 코스 경쟁, 크루 러닝의 상태 전이가 한 흐름에 섞이면 UI 분기와 기록 저장 로직이 복잡해짐 해결: useSoloRunning, useSoloCourseRunning, useCompetitorRunning, useCrewRunning으로 러닝 모드별 로직 분리
결과: 모드별 위치 전송, 거리 계산, 기록 저장 책임을 분리해 유지보수 범위를 줄임

Case 02

문제: 러닝 중 위치 동기화가 네트워크 상태에 의존함

문제: 러닝 중 위치 동기화가 네트워크 상태에 의존함 해결: STOMP 클라이언트에 reconnectDelay, heartbeatIncoming, heartbeatOutgoing을 적용하고 모드별 publish/subscribe 경로 분기
결과: 연결 끊김 후 재연결을 시도하고, 개인 러닝과 크루 러닝의 데이터 흐름을 분리

Case 03

문제: GPS 좌표와 속도 값은 플랫폼과 환경에 따라 흔들릴 수 있음

문제: GPS 좌표와 속도 값은 플랫폼과 환경에 따라 흔들릴 수 있음 해결: 솔로 러닝에서 2초 단위 좌표를 누적하고 최근 10초 거리 기반으로 페이스를 계산
결과: 단일 GPS speed 값에만 의존하지 않고 거리 기반 페이스 표시 흐름을 구성

Case 04

문제: 러닝 전후 위치 추적 정확도 요구가 다름

문제: 러닝 전후 위치 추적 정확도 요구가 다름 해결: 러닝 상태에서는 BestForNavigation, 2초 주기, 1m 간격으로 추적하고 일반 상태에서는 더 낮은 빈도로 전환
결과: 러닝 중 정확도와 평상시 배터리 사용 사이의 기준을 코드로 분리

Case 05

문제: access token 만료 시 여러 요청이 동시에 실패할 수 있음

문제: access token 만료 시 여러 요청이 동시에 실패할 수 있음 해결: Axios 응답 인터셉터에서 refresh 진행 중 요청을 queue에 저장하고 새 토큰 발급 후 재실행
결과: 중복 refresh 요청을 줄이고 인증 실패 복구 흐름을 중앙화

Tech Stack

Why these tools were used.

React Native + Expo

위치 권한, 지도, 배포 흐름을 모바일 앱 기준으로 빠르게 구현하기 위해 사용

TypeScript

러닝 모드, 위치 payload, 기록 데이터 구조를 타입으로 제한하기 위해 사용

Zustand

러닝 상태, 위치, 코스, 인증 정보를 화면 밖에서도 공유해야 해서 사용

TanStack Query

코스, 경쟁자 기록, 크루 데이터처럼 서버 상태를 캐싱하고 조회 상태를 관리하기 위해 사용

WebSocket(STOMP)

러닝 중 위치와 준비 상태를 서버와 실시간으로 주고받기 위해 사용

expo-location

foreground/background 위치 권한과 위치 구독 옵션을 제어하기 위해 사용

React Native Maps

코스 polyline, 현재 위치, 크루/코스 마커를 지도 위에 렌더링하기 위해 사용

Axios

인증 헤더 주입, 401 처리, 토큰 갱신 재요청을 한 계층에서 처리하기 위해 사용

Focus

What I focused on.

  • 위치 기반 앱에서 화면 표시 값과 저장 데이터가 같은 기준으로 흐르도록 상태 책임을 분리
  • 실시간 연결을 컴포넌트 이벤트가 아니라 러닝 모드와 상태 전이에 맞춰 관리
  • 러닝 중 정확도, 배터리 사용, 네트워크 재연결 사이의 기준을 코드로 명확히 표현
  • 백엔드와 STOMP destination, 위치 payload, 경쟁자 기록 조회 방식을 맞춰 데이터 흐름을 정리
Trade-offs

Limits and decisions.

  • 러닝 중 위치 정확도를 높이기 위해 BestForNavigation을 사용했지만 배터리 사용량이 증가할 수 있음
  • 일부 경쟁/크루 러닝 페이스 계산은 아직 GPS raw speed 값에 의존해 플랫폼별 편차 가능성이 남아 있음
  • STOMP 연결과 구독 흐름이 러닝 상태에 강하게 묶여 있어 모드가 늘어나면 공통 추상화가 추가로 필요함
  • 측정 수치가 없는 개선은 구조 개선 중심으로만 정리했고, 성능 향상 수치로 포장하지 않음
Portfolio Summary

How to read this project.

  • 러너스하이는 위치 기반 실시간 러닝을 지원하는 React Native 모바일 앱입니다.
  • FE 1명으로 지도, 코스 탐색, 솔로 러닝, 과거 기록 경쟁, 크루 러닝 화면을 구현했습니다.
  • Zustand로 위치, 러닝, 코스, 인증 상태의 책임을 나누고 러닝 모드별 커스텀 훅으로 상태 전이를 분리했습니다.
  • STOMP 기반 WebSocket으로 러닝 중 위치와 준비 상태를 실시간 동기화했습니다.
  • reconnectDelay와 heartbeat 설정으로 러닝 중 연결 끊김에 대한 복구 흐름을 구성했습니다.
  • 솔로 러닝에서는 최근 10초 거리 기반 페이스 계산으로 GPS speed 직접 의존도를 낮췄습니다.
  • Axios 인터셉터와 refresh queue로 인증 만료 시 요청 재시도 흐름을 공통 처리했습니다.
