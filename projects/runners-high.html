Portfolio Summary

  • 러너스하이는 위치 기반 실시간 러닝을 지원하는 React Native 모바일 앱입니다.
  • FE 1명으로 지도, 코스 탐색, 솔로 러닝, 과거 기록 경쟁, 크루 러닝 화면을 구현했습니다.
  • Zustand로 위치, 러닝, 코스, 인증 상태의 책임을 나누고 러닝 모드별 커스텀 훅으로 상태 전이를 분리했습니다.
  • STOMP 기반 WebSocket으로 러닝 중 위치와 준비 상태를 실시간 동기화했습니다.
  • reconnectDelay와 heartbeat 설정으로 러닝 중 연결 끊김에 대한 복구 흐름을 구성했습니다.
  • 솔로 러닝에서는 최근 10초 거리 기반 페이스 계산으로 GPS speed 직접 의존도를 낮췄습니다.
  • Axios 인터셉터와 refresh queue로 인증 만료 시 요청 재시도 흐름을 공통 처리했습니다.

Highlight Points

Technical Highlights

Case 01
Problem

GPS 신호 편차로 거리와 페이스 계산 값이 흔들렸습니다.

Solution

expo-location 수집 주기와 거리 누적 로직을 조정하고, 위치 업데이트 흐름을 러닝 모드별로 분리했습니다.

Result

거리/페이스 계산 로직을 러닝 기록 저장 흐름과 분리해 GPS 편차가 화면 표시와 기록 저장에 미치는 영향을 줄였습니다.

Case 02
Problem

네트워크 오류나 일시적 연결 끊김이 발생하면 실시간 위치 동기화가 중단됐습니다.

Solution

heartbeatIncoming, heartbeatOutgoing, reconnectDelay를 적용한 STOMP 클라이언트와 모드별 publish/subscribe 분기 구조를 구현했습니다.

Result

주행 중 WebSocket 연결이 끊겼을 때 재연결을 시도하고, 모드별 위치 전송 흐름을 분리해 데이터 흐름을 안정화했습니다.

Case 03
Problem

경쟁자 비교 기능에서 매 초 경쟁자 진행도를 수신해 네트워크 요청과 클라이언트 처리 비용이 커졌습니다.

Solution

백엔드와 경쟁자 기록 전달 방식을 시계열 조회 구조로 조정하고, 프론트엔드에서 현재 러너의 진행도와 비교하도록 변경했습니다.

Result

매 초 수신 구조를 제거해 불필요한 네트워크 요청을 줄이고, 비교 계산 책임을 클라이언트에서 처리하도록 정리했습니다.

What I Focused On

  • 위치 기반 앱에서 화면 표시 값과 저장 데이터가 같은 기준으로 흐르도록 상태 책임을 분리

  • 실시간 연결을 컴포넌트 이벤트가 아니라 러닝 모드와 상태 전이에 맞춰 관리

  • 러닝 중 정확도, 배터리 사용, 네트워크 재연결 사이의 기준을 코드로 명확히 표현

  • 백엔드와 STOMP destination, 위치 payload, 경쟁자 기록 조회 방식을 맞춰 데이터 흐름을 정리
