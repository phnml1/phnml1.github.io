Project Case Study

Fly:On

패러글라이딩 체험장 탐색과 AI 여행 일정 생성을 제공하는 React Native 모바일 앱입니다.

Period2025.06 - 2025.11
TeamFE 2 / BE 2 / Design 1
Role일정 생성/편집 플로우, 지도 기반 체험장 탐색, 인증/에러 처리 일부
React NativeExpoExpo RouterTypeScriptTanStack QueryZustandAxiosReact Native Maps
Overview

What this project solves.

  • 패러글라이딩 체험장 탐색과 AI 여행 일정 생성을 제공하는 React Native 모바일 앱입니다.
  • 사용자가 날짜, 지역, 체험장, 관광지를 단계별로 선택한 뒤 추천 일정을 생성하고 편집할 수 있도록 구성했습니다.
  • 복잡한 일정 편집을 모바일에서 처리하기 위해 커스텀 Drag & Drop, 단계형 상태 관리, 지도 기반 탐색 흐름을 구현했습니다.
Impact

Results worth noticing.

Result 01

React DevTools Profiler 기준 Drag & Drop 시나리오의 total commit duration을 211.5ms -> 72.9ms로 줄였습니다.

Result 02

평균 commit duration을 17.6ms -> 6.6ms로 낮췄습니다.

Result 03

Drag & Drop 로직을 useDragDrop, useAutoScroll, useLayoutMeasurement로 분리해 제스처 처리, 자동 스크롤, 레이아웃 측정 책임을 나눴습니다.

Result 04

단계별 진행 조건을 validateNextStepEnabled 순수 함수로 분리하고 Jest 테스트로 검증했습니다.

Key Features

Product surface.

단계형 일정 생성 플로우

날짜, 지역, 체험장, 관광지 선택을 순차적으로 진행해 사용자가 필요한 입력을 빠뜨리지 않도록 제어했습니다.

AI 추천 일정 생성 및 편집

생성된 추천 일정을 사용자가 직접 순서 변경할 수 있도록 편집 화면을 제공했습니다.

커스텀 Drag & Drop 일정 편집

React Native 환경에서 PanResponder, layout 측정, floating layer를 조합해 일정 카드 이동을 구현했습니다.

지도 기반 체험장 탐색

선택 지역의 좌표 경계를 기준으로 지도 이동 범위를 제한하고, 지역별 체험장 marker를 조회했습니다.

인증 기반 앱 진입 흐름

저장된 토큰을 초기화 단계에서 검증하고, 인증 여부에 따라 접근 가능한 라우트를 분리했습니다.

공통 에러 처리

API 실패와 렌더링 오류를 전역 모달과 Error Boundary로 처리해 화면별 예외 처리 중복을 줄였습니다.

Technical Highlights

Problems, decisions, outcomes.

Case 01

1. Drag & Drop 렌더링 비용 개선

  • 문제: 일정 편집 화면에서 Drag & Drop 중 렌더링 비용이 커져 조작이 무겁게 느껴졌습니다.
  • 해결: React DevTools Profiler로 commit duration을 측정하고, React.memo, Zustand selector, callback/ref 안정화, floating layer 분리를 적용했습니다.
  • 결과: 동일 Drag & Drop 시나리오 기준 total commit duration을 211.5ms -> 72.9ms로 줄였습니다.
  • 근거: flyon/profiler json/성능개선전.json, flyon/profiler json/렌더링 성능개선후(2).json
Case 02

2. React Native 커스텀 Drag & Drop 구현

  • 문제: React Native에서는 웹의 Drag & Drop API를 사용할 수 없어 일정 카드 편집 인터랙션을 직접 구현해야 했습니다.
  • 해결: PanResponder로 제스처를 처리하고, useLayoutMeasurement로 Day/card 위치를 측정해 드롭 위치를 계산했습니다.
  • 결과: 같은 날짜 내 순서 변경과 다른 날짜로의 카드 이동을 모바일 화면에서 처리할 수 있게 했습니다.
  • 근거: DraggablePlanCard.tsx, useDragDrop.ts, useLayoutMeasurement.ts
Case 03

3. Drag layer 분리로 레이어 충돌 해결

  • 문제: 리스트 내부에서 카드를 직접 움직이면 드래그 카드가 다른 UI 아래로 가려지는 문제가 발생했습니다.
  • 해결: FloatingPortal을 만들어 드래그 중인 카드를 별도 absolute layer에서 렌더링했습니다.
  • 결과: 드래그 중인 카드가 리스트 구조와 분리되어 화면 위에서 안정적으로 표시되도록 만들었습니다.
  • 근거: FloatingPortal.tsx, DraggablePlanCard.tsx
Case 04

4. 단계형 플로우 상태 관리

  • 문제: 날짜, 지역, 체험장, 관광지 선택 상태가 여러 화면에 걸쳐 공유되어 상태 흐름이 복잡해질 수 있었습니다.
  • 해결: useScheduleStore로 현재 단계와 선택값을 관리하고, validateNextStepEnabled로 단계별 진행 조건을 분리했습니다.
  • 결과: 화면 컴포넌트는 입력과 렌더링에 집중하고, 진행 가능 여부는 테스트 가능한 순수 함수로 관리했습니다.
  • 근거: useScheduleStore.ts, validateNextStepEnabled.ts, validateNextStepEnabled.test.ts
Case 05

5. 서버 상태와 클라이언트 상태 분리

  • 문제: 서버에서 가져온 일정/관광지 데이터와 사용자가 편집 중인 임시 상태가 섞이면 캐싱과 UI 편집 흐름이 복잡해질 수 있었습니다.
  • 해결: 조회 데이터는 TanStack Query로 관리하고, 일정 생성 단계/드래그/지도 선택 상태는 Zustand로 분리했습니다.
  • 결과: 서버 데이터 캐싱 정책과 사용자 인터랙션 상태를 각각 다른 책임으로 관리했습니다.
  • 근거: useTourSchedule.ts, usePlaceList.ts, useGptSchedule.ts, useScheduleStore.ts, useDragStore.ts, exploreStore.ts
Case 06

6. 지도 탐색 UX 제어

  • 문제: 지도에서 사용자가 선택 지역 밖까지 이동하면 체험장 탐색 맥락이 흐려질 수 있었습니다.
  • 해결: 선택 지역 좌표에서 northEast, southWest 경계를 계산하고 fitToCoordinates, setMapBoundaries를 적용했습니다.
  • 결과: 사용자가 선택한 지역 범위 안에서 체험장 marker를 탐색하도록 흐름을 제한했습니다.
  • 근거: ExploreMapView/index.tsx, ExploreSpotMarkers.tsx, useSpots.ts, exploreStore.ts
Tech Stack

Why these tools were used.

React Native / Expo

모바일 앱 화면, 네이티브 API 접근, Expo Router 기반 라우팅을 빠르게 구성하기 위해 사용했습니다.

Expo Router

파일 기반 라우팅과 Stack.Protected를 이용해 인증 상태에 따른 화면 접근을 분리했습니다.

TypeScript

일정, 장소, 드래그 카드, API 응답 타입을 명확히 관리하기 위해 사용했습니다.

TanStack Query

일정, 관광지, 날씨, 체험장 조회 데이터에 캐싱과 요청 상태 관리를 적용하기 위해 사용했습니다.

Zustand

일정 생성 단계, 선택값, 드래그 상태, 지도 선택 marker처럼 화면 간 공유되는 클라이언트 상태를 가볍게 관리하기 위해 사용했습니다.

React Native Maps

지역 기반 체험장 탐색과 marker 표시를 구현하기 위해 사용했습니다.

Axios

공통 API client, 인증 헤더 주입, 응답/에러 interceptor를 구성하기 위해 사용했습니다.

React Hook Form / Zod

로그인/회원가입 입력값 검증을 폼 로직과 분리하기 위해 사용했습니다.

Jest

단계별 진행 조건, 인증 store, 일정 조회 hook, 일정 데이터 변환 로직을 검증하기 위해 사용했습니다.

Focus

What I focused on.

  • 모바일에서 조작 가능한 일정 편집 경험을 만드는 데 집중했습니다.
  • 드래그 중 카드가 가려지지 않도록 렌더링 계층을 분리했습니다.
  • 서버 데이터와 사용자가 편집 중인 상태를 분리해 캐싱 정책과 UI 상태가 섞이지 않도록 했습니다.
  • 단계형 플로우의 진행 조건을 화면 밖으로 분리해 테스트 가능한 구조로 만들었습니다.
  • 지도 탐색에서는 사용자가 선택한 지역 안에서만 체험장을 탐색하도록 boundary를 제어했습니다.
Trade-offs

Limits and decisions.

  • Drag & Drop을 직접 구현하면서 제스처 처리, 레이아웃 측정, 자동 스크롤 로직이 늘어났습니다.
  • PanResponder 기반 구현은 세밀한 제어가 가능하지만, Reanimated 기반 구현보다 애니메이션 최적화 여지가 남아 있습니다.
  • GPT 일정 생성 hook은 retry: 30, retryDelay: 2000으로 응답 지연을 고려했지만, 사용자 취소/재시도 UX는 더 명확히 개선할 수 있습니다.
  • fetchGptSchedule에서 편집용 clientKey를 Math.random()으로 생성하고 있어, 장기적으로는 서버 ID 또는 deterministic key가 더 안정적입니다.
  • WebView 기반 3D 트랙과 비행 기록 기능은 프로젝트 전체 기능이지만, 본인 담당 성과로는 강조하지 않습니다.
Portfolio Summary

How to read this project.

Fly:On은 패러글라이딩 체험장 탐색과 AI 여행 일정 생성을 제공하는 React Native 모바일 앱입니다. 저는 일정 생성/편집 플로우와 지도 기반 체험장 탐색 기능을 중심으로 개발했습니다. 일정 편집 화면에서는 PanResponder, layout 측정, floating layer를 조합해 커스텀 Drag & Drop을 구현했습니다. React DevTools Profiler로 렌더링 병목을 측정하고, total commit duration을 211.5ms -> 72.9ms로 줄였습니다. 일정 생성 단계는 Zustand로 관리하고, 단계별 진행 조건은 순수 함수와 테스트로 분리했습니다. 지도 탐색에서는 선택 지역 boundary를 제한하고, 지역별 marker 조회에 TanStack Query 캐싱을 적용했습니다. API 에러와 렌더링 오류는 전역 모달과 Error Boundary로 처리해 예외 상황의 사용자 흐름을 정리했습니다.
